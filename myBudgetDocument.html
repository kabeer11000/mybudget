myBudget - Complete System Architecture

System Overview

myBudget is a monolithic personal finance tracking app with AI-powered natural language transaction entry. Users can track expenses, income, set budgets, and view analytics.

Tech Stack:

  • Frontend: React Native (Expo)
  • Backend: Node.js/Bun (monolithic)
  • Database: MongoDB/Firestore
  • Auth: Firebase Auth
  • AI: Gemini + NLP

1. Authentication Flows

1.1 User Signup

sequenceDiagram
    actor User
    participant System
    participant UserDomain
    User->>System: Selects "Sign Up"
    System->>User: Display signup form (email, password, confirm password)
    
    User->>System: Fills in details and submits
    System->>System: Validate input (email format, password rules)
    
    alt Invalid input
        System->>User: Show error, request correction
    else Valid input
        System->>UserDomain: createUser(email, password)
        UserDomain->>UserDomain: Check if email exists
        
        alt Email already exists
            UserDomain->>System: Email exists error
            System->>User: "Email already registered"
        else Email available
            UserDomain->>UserDomain: Hash password & create account
            UserDomain->>System: User created, return token
            System->>User: "Account created successfully"
            System->>User: Redirect to home screen
        end
    end

1.2 User Login

sequenceDiagram
    actor User
    participant System
    participant UserDomain

    User->>System: Selects "Login"
    System->>User: Display login form<br/>(email, password)
    
    User->>System: Enters credentials and submits
    System->>UserDomain: loginUser(email, password)
    
    alt Invalid credentials
        UserDomain->>System: Authentication failed
        System->>User: "Invalid email or password"<br/>Offer "Forgot Password"
    else Valid credentials
        UserDomain->>UserDomain: Generate token<br/>Update last login
        UserDomain->>System: Login successful, return token
        System->>User: Redirect to home screen
    end

1.3 Forgot Password

sequenceDiagram
    actor User
    participant System
    participant UserDomain
    participant EmailService

    User->>System: Clicks "Forgot Password"
    System->>User: Display email input
    
    User->>System: Enters email and submits
    System->>UserDomain: resetPasswordRequest(email)
    UserDomain->>UserDomain: Check if user exists
    
    Note over UserDomain: Always return success<br/>(security: hide user existence)
    
    alt User exists
        UserDomain->>UserDomain: Generate 6-digit code
        UserDomain->>UserDomain: Store code with expiry (15 min)
        UserDomain->>EmailService: Send reset email with code
    end
    
    UserDomain->>System: Success response
    System->>User: "Check your email for reset code"
    
    User->>System: Enters code and new password
    System->>UserDomain: resetPasswordConfirm(email, code, newPassword)
    
    alt Code invalid or expired
        UserDomain->>System: Validation error
        System->>User: "Invalid or expired code"<br/>Allow retry
    else Code valid
        UserDomain->>UserDomain: Hash new password<br/>Update user<br/>Clear reset code<br/>Clear all sessions
        UserDomain->>System: Password reset successful
        System->>User: "Password reset successful"
        System->>User: Redirect to login
    end

1.4 Change Password

sequenceDiagram

    else Current password correct

        UserDomain->>System: Password changed
        System->>User: "Password changed successfully"
    end

3. Budget & Analytics Flows

3.1 Add Monthly Budget

sequenceDiagram
    actor User
    participant System
    participant BudgetDomain

    System->>User: "Budget set successfully"
    System->>User: Show budget progress

3.2 Track Budget Progress

sequenceDiagram
    actor User
    participant System
    participant BudgetDomain
    participant TransactionDomain

    User->>System: Opens Budget Progress screen
    System->>BudgetDomain: getBudget(userId, month, year)
    
    alt No budget set
        BudgetDomain->>System: No budget found
        System->>User: "No budget set"<br/>Show "Add Budget" button
    else Budget exists
        BudgetDomain->>System: Budget amount
        System->>TransactionDomain: getTotalExpenses(userId, month, year)
        TransactionDomain->>System: Total spent
        System->>System: Calculate:<br/>- remaining<br/>- percentage used
        System->>User: Display progress bar<br/>and statistics
        
        alt Overspending detected
            System->>User: Show warning alert
        end
    end

3.3 Get Overspending Alerts

sequenceDiagram
    actor User
    participant System
    participant BudgetDomain
    participant TransactionDomain
    participant NotificationService

    Note over System: Triggered after adding expense

    System->>BudgetDomain: getBudget(userId, month, year)
    BudgetDomain->>System: Budget amount
    System->>TransactionDomain: getTotalExpenses(userId, month, year)
    TransactionDomain->>System: Total spent
    
    alt Spending exceeds budget
        System->>System: Calculate overage
        System->>NotificationService: sendOverspendingAlert(userId, overage)
        NotificationService->>System: Alert sent
        System->>User: Display overspending notification<br/>"You've exceeded your budget by X"
    else Within budget
        Note over System: No action needed
    end

5. API Endpoints Summary

Authentication

Method Endpoint Request Body Response
POST /auth/signup {email, password} {userId, token}
POST /auth/login {email, password} {userId, token}
POST /auth/logout {token} {success}
POST /auth/password-reset {email} {success}
POST /auth/password-reset/confirm {email, resetCode, newPassword} {success}
PUT /auth/password {currentPassword, newPassword} {success}
PUT /users/profile {firstName, lastName, phone} {user}

Transactions

Method Endpoint Request Body Response
POST /transactions/expense {amount, category, description, date} {transaction}
POST /transactions/income {amount, category, description, date} {transaction}
POST /transactions/parse {text} {type, amount, category, description, date}
GET /transactions?month=X&year=Y - {transactions[]}
PUT /transactions/:id {amount, category, description, date} {transaction}
DELETE /transactions/:id - {success}

Budget

Method Endpoint Request Body Response
POST /budget {amount, month, year} {budget}
GET /budget?month=X&year=Y - {budget, spent, remaining, percentage}

Method Endpoint Request Body Response

GET /analytics/categories?month=X&year=Y - {categories[]}

6. Use Case Descriptions

UC-1: User Signup

Actors: User
Preconditions: User is not already registered
Postconditions: New user account created and user logged in
Flow:

  1. User selects “Sign Up”
  2. System displays signup form
  3. User fills in email and password
  4. System validates input
  5. If valid, system creates account and generates token
  6. User receives confirmation and is redirected to home screen

Exception Conditions:

  • Email already exists → notify user
  • Password doesn’t meet requirements → request correction
  • Missing fields → request completion

UC-2: User Login

Actors: User
Preconditions: User has registered account
Postconditions: User is logged in
Flow:

  1. User selects “Login”
  2. System displays login form
  3. User enters email and password
  4. System validates credentials
  5. If correct, system redirects to home screen

Exception Conditions:

  • Invalid credentials → allow retry
  • User can click “Forgot Password”

UC-3: Forgot Password

Actors: User, Email Service
Preconditions: User has valid registered email
Postconditions: Password is updated
Flow:

  1. User selects “Forgot Password”
  2. System asks for email
  3. User provides email
  4. System sends reset code via email
  5. User enters code and new password
  6. System updates password

Exception Conditions:

  • Email not found → notify user (but don’t reveal)
  • Reset code expires → allow retry

UC-4: Change Password

Actors: User
Preconditions: User is logged in
Postconditions: Password is changed
Flow:

  1. User selects “Change Password”
  2. System shows password form
  3. User enters current and new password
  4. System validates current password
  5. If valid, updates password

Exception Conditions:

  • Current password incorrect → notify user
  • New passwords don’t match → request correction

UC-5: Logout

7. Security & Performance

Security Measures

  • Password Hashing: bcrypt with salt rounds ≥ 10
  • JWT Tokens: Short expiry (15-30 minutes)
  • Session Management: Database-stored, invalidation on password change
  • Rate Limiting: 5 auth attempts/15 min, 100 API requests/min
  • Input Validation: Server-side validation, parameterized queries
  • HTTPS Only: All communication encrypted

Performance Considerations

  • System responds within 30 seconds (standard timeout)
  • Statistical operations may take longer
  • Real-time data processing requires adequate hardware
  • Database: MongoDB/Firestore handles concurrent requests
  • Offline capability planned for future iterations
